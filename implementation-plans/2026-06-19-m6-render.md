## 1. Goal

Translate IR into the **natural-language instruction text** that lands in generated
markdown: numbered runbook steps, subagent prompt bodies (with the §9 output protocol
appended), and neutral frontmatter *values*. This is the host-agnostic **"what to say"**
layer. Bindings (M7/M10) decide **"where to put it"** (file layout, frontmatter keys,
tool names).

The seam between render and binding is the `Vocabulary` interface: render calls
`vocab.InvokeAgent(...)` etc., and each host supplies phrasing. Render owns **control-flow
prose and the output protocol**; the host owns **verbs and tool references**.

## 2. Deliverables

- `internal/render/vocabulary.go` — `Vocabulary` interface + a `Default`/test impl.
- `internal/render/render.go` — top-level `Document` builder + entry points.

## 3. The `Vocabulary` seam (`internal/render/vocabulary.go`)

```go
package render

type Vocabulary interface {
    // Verb phrases for kernel actions. Each returns a sentence/clause.
    InvokeAgent(a AgentView) string                 // "Invoke the `reviewer` subagent…"
    RunScript(g GateView) string                    // "Run `scripts/test.sh`."
    SpawnParallel(branches []StepView) string       // "Spawn these subagents together…"
    ReadOutput(valueLabel string) string            // "Read the `out:` value from `review`."
    ParseOutputProtocol(enum []string, retry int) string // §9.3 instructions
    GotoStep(controlLabel string) string            // "Go back to step `build`."
    Arg(name string) string                         // host token for flow input, e.g. "$ARGUMENTS"
}
```

- `AgentView`/`GateView`/`StepView` are small read-only views over `ir` nodes (define in
  `render` to avoid leaking the whole IR into vocab impls).
- Render provides a `defaultVocabulary` used by render's own unit tests so M6 is testable
  without M7. M7 supplies `claudeVocabulary`.
- **Rule:** every host-specific noun/verb/tool-name flows through `Vocabulary`. Render
  never hardcodes "Task tool", "$ARGUMENTS", `.claude/...`, etc.

## 4. Output `Document` (`internal/render/render.go`)

```go
type Document struct {
    Frontmatter []FMField // ordered key/value pairs (binding maps to host format)
    Body        string    // rendered markdown body
}
type FMField struct{ Key, Value string }
```

Top-level entry points:

```go
// AgentDocument renders one subagent's instruction file body + neutral frontmatter
// values (name/description/tools/model-alias). Binding decides frontmatter keys/format.
func AgentDocument(p ir.Program, a ir.Agent, v Vocabulary) Document

// RunbookDocument renders the entry flow as a numbered runbook (the command file body).
func RunbookDocument(p ir.Program, v Vocabulary) Document
```

`Document.Frontmatter` holds **neutral values** (e.g. `Key:"model-alias", Value:"opus"`);
the binding renames to `model: claude-opus-...` and formats YAML. Render does not emit YAML
syntax — it hands structured fields up.

## 5. Runbook rendering (`internal/render/runbook.go`)

Walk `ir.Flow.Root`, emitting **numbered steps** in `ir.Flow.Order`. Each kernel node has a
renderer. Output is deterministic markdown.

### 5.1 Step numbering

Use a flat incrementing counter for top-level sequence steps; nested constructs
(loop/branch bodies) use sub-numbering (`3a`, `3b`) or indented bullet lists —
**decision: numbered top-level steps + indented sub-steps** under control constructs, with
the control-label shown in backticks so gate targets are unambiguous. Pin exact format in
goldens.

### 5.2 Renderers per kernel construct

| Construct | Rendering (host-neutral; verbs via Vocabulary) |
|-----------|-----------------------------------------------|
| **Sequence** | Numbered steps. When a step consumes a prior producer, append "using the output from `<prevProducer>`" (from `StepRef.PrevProducer`, spec §4.5). |
| **Agent leaf** | `vocab.InvokeAgent(view)` + pass input (flow input via `vocab.Arg` if `In` matches entry input; sequential context otherwise). |
| **Gate leaf** | `vocab.RunScript(view)`. On failure, render the `OnFail` policy prose (§5.3). |
| **Branch** | "Read the `out:` value from `<branchValue>` (`vocab.ReadOutput`). Then: " followed by one bullet per case: "if `review` is `approve`, do step …". No dotted paths (v0.1). |
| **Loop (while)** | "Repeat the following steps while `<cond>` is **not** satisfied, at most `N` times (advisory — count your iterations):" then the body. `until X != Y` → "until `X` equals `Y`". Advisory wording per §12. |
| **Repeat (do-while)** | Render the body **once**, then: "Then repeat the steps above until `<cond>`, at most `N` times. The check happens **after** each pass, so always run the steps at least once." (spec §4.7.1 — must make first-pass explicit.) |
| **Parallel/gather** | `vocab.SpawnParallel(branches)`; then gather step receives the payload: "Collect the outputs from `lint`, `security`, `style` and give them to `reviewer`." Plus sequential context from the producer before the parallel. |

### 5.3 Gate failure prose (§7.4.1)

Render explicit, no-bounce-back wording keyed on `OnFail`:

- `halt`: "If the gate fails, stop the flow and report failure."
- `retry` + target: "If the gate fails, go back to step `<target>` and re-run from there.
  Retry the script up to `<scriptRetry>` times first." → uses `vocab.GotoStep(target)`.
- `goto` + target: "If the gate fails, jump to step `<target>`."
- `enter-loop`: "If the gate fails, re-enter the current loop."

Never emit the word "bounce-back".

### 5.4 Loop/repeat bound wording (§12)

Native runbook is **advisory** for loop bounds. Always include "(advisory — track your
iteration count and stop at N)" so the runbook honestly reflects the native tier. The
hook-enforced/SDK tiers tighten this later; render's job is honest native phrasing.

## 6. Prompt rendering (`internal/render/prompt.go`)

```go
func AgentPrompt(a ir.Agent, v Vocabulary) string
```

1. Start from `a.Prompt` — the **already-resolved** base text from IR (M2 read any file;
   M5 carried it). Render **never touches the filesystem**; file- and inline-sourced
   prompts render identically.
2. If `a.Out` is an enum (`len(a.OutEnum) > 0`), append the §9 output-protocol block
   (§7).
3. Leave `${NAME}` **unexpanded** (bindings handle env at emit time).

## 7. Output protocol text (`internal/render/protocol.go`, spec §9)

```go
func OutputProtocol(enum []string, retry int) string
```

Emit exactly (spec §9.1), with a real fenced block using the literal tag
`agentflow-output`:

```
When you have finished, end your reply with exactly one fenced block in this form
(no other text after the closing fence):

```agentflow-output
out: <value>
```

`<value>` must be exactly one of: approve, revise, reject.
```

- Fence tag is **exactly** `agentflow-output`.
- The single line is always `out: <value>` (field name always `out` in v0.1, §9.1).
- Enum members listed comma-separated in declaration order.
- The **runbook** (not the prompt) documents the parse/retry/halt table (§9.3) via
  `vocab.ParseOutputProtocol(enum, retry)`: "Read the last `agentflow-output` block. If it
  is missing or invalid, re-invoke the agent up to `<retry>` times, then stop the flow."

> Escaping the inner fence inside a Go string: produce the literal backtick fences; in
> tests, store the golden as a raw file so the nested code fence is exact.

## 8. Testing

Goldens: `internal/render/testdata/*.txt` (or `.md`). Use render's `defaultVocabulary` so
tests are host-independent. Test file `internal/render/render_test.go`.

### 8.1 Per-construct golden snapshots

One focused golden per construct using minimal IR built in-test (or sliced from fixtures):
sequence, branch, loop, repeat, parallel/gather, gate (each `OnFail` variant), output
protocol.

### 8.2 Fixture-level goldens

- `reviewer` prompt golden: base text + `agentflow-output` block listing
  `approve|revise|reject`.
- `critic.af` repeat runbook: body rendered once, then "repeat … until `verdict` equals
  `pass` (max 3)" with the after-the-body note.
- `review.af` runbook golden: gate retry targets control label `build`; loop reads `review`
  `out:`; branch on `code_review` reads `Verdict`; gather lists lint/security/style.

### 8.3 Disk-free assertion

A test that renders `reviewer` (whose prompt came from a file in M2) **after deleting the
source file from a temp tree** still produces identical output — proving render never reads
disk. (Construct the IR with the prompt text already inlined, then assert no file access.)

## 9. Acceptance criteria

- [x] Golden text per construct (seq/branch/loop/repeat/parallel/gate/protocol).
- [x] `critic.af` repeat runbook renders body once then the repeat instruction (after-body
      check explicit).
- [x] `reviewer` prompt golden includes the exact `agentflow-output` block with the three
      enum members.
- [x] `review.af` runbook golden: gate `retry` → control label `build`; loop on

- [x] All host-specific phrasing goes through `Vocabulary` (no hardcoded host nouns in
      render — enforce by review + a grep-style test for forbidden tokens like `.claude`).
- [x] Render performs zero filesystem access.

## 11. Risks & notes

- **Runbook phrasing is runtime behavior** for the native tier (§12) — small wording
  changes alter what the host LLM does. Pin everything with goldens; review wording with
  care; treat phrasing changes as semver-relevant.
- **Nested code fences** (the `agentflow-output` block inside a markdown body) are the most
  fragile bit to get byte-exact. Keep the protocol block in `protocol.go` as a single
  source and store its golden in a raw file.
- **Vocabulary leakage** is the main design risk. If render finds itself needing a host
  fact not in `Vocabulary`, extend the interface rather than special-casing — otherwise
  Cursor (M10) diverges.
- Keep `Document.Frontmatter` values neutral; do not pre-format YAML. Binding owns
  serialization so Claude and Cursor can differ.
